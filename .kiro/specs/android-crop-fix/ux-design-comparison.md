## 0. 共通前提

- アプリの目的: 「好きな部分をパズル化すること」
- 画面名: ベストショット調整（crop screen）
- 自由版: crop なし。元画像をそのままパズル化。
- Plus版: crop あり。選んだ範囲だけをパズル化。

## 2. 固定比率フレーム方式（3:4案）のUI

### 設計思想

「切り抜く領域の比率」を固定し、ユーザーは「どこを切り抜くか」だけを操作する。
画面全面にオーバーレイを敷き、その中央に固定サイズの透明フレームを表示。
フレーム外 = 暗くなる。フレーム内 = 出力範囲。

### Portrait 393×851

フレームサイズ計算:
- 幅: 393px × 88% = 346px
- 高さ: 346 × 4/3 = 461px（全て wrap 649px 内に収まる）
- 上下余白: (649 - 461) / 2 = 94px

## 3. ユーザー操作フロー比較

### 現設計（全画面）

```
[写真を選択]
    ↓
crop画面が開く
    ↓
画像がcanvas全体を cover で表示される
（画面が黒い枠なし・全面が出力範囲）
    ↓
ユーザーが指で移動・ピンチズーム
    ↓
「この画面の見えてる部分全体」= 出力範囲
    ↓
[この範囲で決定]
    ↓
パズル開始（出力の縦横比 = 画面の縦横比）
```

ユーザーの認知モデル: 「画面の中で画像を動かす → 画面に収まった部分が出力」

### 固定比率フレーム方式

```
[写真を選択]
    ↓
crop画面が開く
    ↓
画像が表示 + 中央にフレーム + フレーム外暗い
    ↓
ユーザーが指で移動・ピンチズーム
「このフレームに入れたい部分を合わせる」
    ↓
「フレームの中だけ」= 出力範囲
    ↓
[この範囲で決定]
    ↓
パズル開始（出力の縦横比 = フレームの縦横比 = 常に3:4）
```

ユーザーの認知モデル: 「フレームの中に入れる → フレーム部分が出力」

## 5. 無料版との整合性

### 現設計での問題

| 版 | 入力 | 出力 | 比率 |
|----|------|------|------|
| 無料版 | 元画像（crop なし） | 元画像 | 元画像の縦横比（可変） |
| Plus版・縦画面 | cropした画像 | wrap比率（≈0.606） | ほぼ元画像に近い |
| Plus版・横画面（現在） | cropした画像 | wrap比率（4.46）| **深刻な歪み** |

Plus版の横画面だけが異常。

### 3:4固定フレーム後の状況

| 版 | 出力比率 | 縦横 |
|----|---------|------|
| 無料版（縦画像） | 元画像比率（例: 0.75） | 縦長 |
| 無料版（横画像） | 元画像比率（例: 1.33） | 横長 |
| Plus版（縦画面） | 3:4 = 0.75 | 縦長 |
| Plus版（横画面） | 3:4 = 0.75 | 縦長 |

評価:
- Plus版は向きによらず常に3:4 → 一貫性あり
- 無料版は元画像依存（横長もある）→ 依然として非一貫
- ただし「Plus版だけ向きで変わる」問題は解消される

### 整合性の本質的な問い

無料版と Plus版の「出力比率の一貫性」を求めるなら、
設計上の選択肢は以下のみ:

```
a) 元画像比率を維持する crop（contain fit → パディング追加）
   → 「好きな部分を選ぶ」がしにくい（全体が見えてしまう）

b) 固定比率 crop（3:4 等）
   → 比率は固定。元画像の縦横は無視。「どこを選ぶか」に集中できる

c) 比率制約なし（現設計の正常化 = FIX-1のみ）
   → 画面向きで比率が変わる。横画面は「横長パズル」
```

## 6. 「好きな部分をパズル化する」目的への適合評価

### 現設計（FIX-1のみ）

```
+ 操作が直感的（画面全体が選択範囲）
+ ユーザーが画像の任意の部分を選べる
+ 実装変更が最小
- 向きを変えると切り抜き形状が変わることをユーザーが理解できない
- 横画面では使えない（改善しても 2.85:1 と歪んでいる）
- 「横画面でパズルを作ると横長になる」ことの告知がない
```

### 3:4固定フレーム

```
+ 「どこを切り抜くか」だけを考えればよい（比率は自動）
+ 向きを変えても出力は常に同じ比率
+ フレームが明確なので誤解が少ない
- 縦画面でも若干余白が生じる（操作領域が減る）
- 横画面ではフレームが非常に小さくなる
- 「横画面では使いにくい」ことを UI で伝える必要がある
- 実装追加が必要（オーバーレイ、フレームDOM、transform計算変更）
```

## 7. 第3の選択肢：「横画面を禁止 or 誘導」案

固定比率を採用しつつ、横画面では明示的に誘導する:

```
横画面でcrop画面を開くと:

┌──────────────────────────────────────────────────────────┐
│                                                          │
│                                                          │
│          📱 端末を縦に持ち直してください                    │
│          ベストショット調整は縦画面専用です                  │
│                                                          │
│            [横画面のまま続ける]   [戻る]                   │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

| 観点 | 評価 |
|------|------|
| UX明確性 | ○（ユーザーが混乱しない） |
| 横画面ユーザーへの強制 | △（「横画面で続ける」を選んだ場合の扱いが必要） |
| 実装コスト | 低（orientation検知 + ダイアログのみ） |
| 「好きな部分を選ぶ」への影響 | なし（縦画面での体験は変わらない） |

## 8. 推奨案の整理

### 判断マトリクス

| 方針 | 出力比率 | 横画面対応 | 実装コスト | 「好きな部分を選ぶ」への適合 |
|------|---------|-----------|-----------|--------------------------|
| FIX-1のみ（現設計正常化） | 向き依存（2.85〜0.60） | 動作するが横長 | 最低 | 高（直感的） |
| 固定3:4フレーム | 常に3:4 | 小さいフレーム | 中 | 高（フレーム明確） |
| 縦画面誘導 + 全画面crop | 縦画面のみ（0.60固定） | 警告表示 | 低 | 高（縦画面のみで解決） |
| 縦画面誘導 + 3:4フレーム | 常に3:4 | 警告表示 | 中 | 最高（比率固定＋フレーム明確） |

### 推奨：「縦画面誘導 + 全画面 crop」

理由:
1. 現設計の「画面全体が出力」という直感的操作を維持
2. 横画面の破損問題を根本から回避（横画面では使わせない or 警告）
3. 固定比率フレームより実装コストが低い
4. 縦画面での比率（≈0.60）は元画像の縦写真に近く自然

次点：「3:4フレーム + 縦画面誘導」（比率の完全な一貫性を求める場合）
